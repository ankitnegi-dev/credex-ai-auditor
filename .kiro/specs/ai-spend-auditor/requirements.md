# Requirements Document

## Introduction

The AI Spend Auditor is a web application for startup founders to audit their AI tool subscriptions. Users input their current AI tool subscriptions via a form, and the audit engine compares what they pay against the cheapest viable plan for their seat count. Results are displayed on a shareable page with a unique UUID, include a Claude-generated personalized summary, and can be emailed to the user via Resend. All audit data and email addresses are persisted in Supabase.

## Glossary

- **Auditor**: The AI Spend Auditor web application.
- **Audit**: A single evaluation session containing one or more tool subscriptions submitted by a user.
- **Audit_ID**: A UUID v4 uniquely identifying an audit record stored in Supabase.
- **Tool_Entry**: A single row in the audit form representing one AI tool subscription (tool name, plan type, seat count, monthly cost).
- **Audit_Engine**: The server-side module that computes optimized plans and calculates savings for each Tool_Entry.
- **Pricing_Catalog**: The static data structure containing the 15 supported AI tools, their available plan tiers, per-seat pricing, and minimum seat thresholds.
- **Optimized_Plan**: The cheapest plan tier in the Pricing_Catalog that satisfies the user's seat count for a given tool.
- **Savings**: The difference between the user's reported monthly cost and the Optimized_Plan cost for the same seat count.
- **Summary**: A 100-word personalized text generated by the Anthropic Claude API describing the audit findings.
- **Report_Page**: The shareable results page rendered at `/audit/[id]`.
- **Resend**: The third-party email delivery service used to send audit reports.
- **Supabase**: The hosted PostgreSQL database and backend-as-a-service used for data persistence.

## Requirements

### Requirement 1: Audit Submission Form

**User Story:** As a startup founder, I want to fill in a form listing my AI tool subscriptions, so that the Auditor can evaluate my spending.

#### Acceptance Criteria

1. THE Auditor SHALL render a subscription input form on the home page (`/`).
2. THE Auditor SHALL provide a dropdown of exactly 15 AI tools: ChatGPT, Claude, Cursor, GitHub Copilot, Midjourney, Gemini, Perplexity, Notion AI, Grammarly, Jasper, Copy.ai, Runway, ElevenLabs, Synthesia, and Otter.ai.
3. THE Auditor SHALL provide a plan type selector with the options: Free, Pro, Team, and Enterprise.
4. THE Auditor SHALL provide a numeric input for number of seats with a minimum value of 1.
5. THE Auditor SHALL provide a numeric input for monthly cost in USD with a minimum value of 0.
6. THE Auditor SHALL allow users to add multiple Tool_Entries to a single audit, with a minimum of 1 and a maximum of 15 entries.
7. WHEN a user submits the form with fewer than 1 Tool_Entry, THE Auditor SHALL display a validation error message and prevent form submission.
8. WHEN a user submits the form with a seat count less than 1 on any Tool_Entry, THE Auditor SHALL display a validation error message on that entry and prevent form submission.
9. WHEN a user submits the form with a monthly cost less than 0 on any Tool_Entry, THE Auditor SHALL display a validation error message on that entry and prevent form submission.
10. WHEN a user submits the form with a missing tool name or plan type on any Tool_Entry, THE Auditor SHALL display a validation error message on that entry and prevent form submission.
11. THE Auditor SHALL display an "Add Tool" button that appends a new empty Tool_Entry row to the form, up to the maximum of 15 entries.
12. WHEN the maximum of 15 Tool_Entries has been reached, THE Auditor SHALL disable the "Add Tool" button.
13. THE Auditor SHALL display a "Remove" control on each Tool_Entry row that removes that row from the form, provided at least 1 entry remains.

### Requirement 2: Audit Engine — Savings Calculation

**User Story:** As a startup founder, I want the Auditor to compare what I pay against the cheapest viable plan, so that I can see exactly where I am overspending.

#### Acceptance Criteria

1. WHEN a valid audit form is submitted, THE Audit_Engine SHALL look up each Tool_Entry in the Pricing_Catalog.
2. WHEN a Tool_Entry is found in the Pricing_Catalog, THE Audit_Engine SHALL identify the Optimized_Plan as the lowest-cost plan tier whose minimum seat threshold is less than or equal to the Tool_Entry's seat count; IF multiple plan tiers share the same lowest cost, THE Audit_Engine SHALL select the tier with the highest minimum seat threshold among those ties.
3. THE Audit_Engine SHALL calculate per-tool Savings as: `round(user_monthly_cost − (optimized_plan_price_per_seat × seat_count), 2)`, where monetary values are rounded to 2 decimal places.
4. IF the calculated per-tool Savings is less than 0, THEN THE Audit_Engine SHALL set that tool's Savings to 0.00 (the user is already on an optimal or better plan).
5. WHEN a valid audit form is submitted, THE Audit_Engine SHALL calculate total monthly spend as the sum of all Tool_Entry monthly costs, rounded to 2 decimal places.
6. WHEN a valid audit form is submitted, THE Audit_Engine SHALL calculate total optimized spend as the sum of all per-tool `optimized_plan_price_per_seat × seat_count` values, rounded to 2 decimal places.
7. WHEN a valid audit form is submitted, THE Audit_Engine SHALL calculate total Savings as the sum of all per-tool Savings values, rounded to 2 decimal places.
8. WHEN a Tool_Entry's tool name is not found in the Pricing_Catalog, THE Audit_Engine SHALL set that tool's Optimized_Plan to the user's reported plan and Savings to 0.00.
9. WHEN a Tool_Entry's tool name is found in the Pricing_Catalog but no plan tier has a minimum seat threshold less than or equal to the Tool_Entry's seat count, THE Audit_Engine SHALL fall back to the plan tier with the highest minimum seat threshold for that tool and set Savings to 0.00.

### Requirement 3: Audit Persistence

**User Story:** As a startup founder, I want my audit results to be saved with a unique link, so that I can revisit or share them later.

#### Acceptance Criteria

1. WHEN the Audit_Engine completes a calculation, THE Auditor SHALL generate a UUID v4 as the Audit_ID.
2. WHEN the Audit_Engine completes a calculation, THE Auditor SHALL persist the following to Supabase in a single atomic write: Audit_ID, all Tool_Entries (tool name, plan type, seat count, monthly cost), per-tool Optimized_Plan and Savings, total spend, total optimized spend, total Savings, and the created-at timestamp.
3. IF the Supabase write fails, THEN THE Auditor SHALL return an error response with HTTP status 500, a human-readable error message describing the failure, and SHALL NOT partially persist any audit data (the write is rolled back or not attempted).
4. WHEN the audit is successfully persisted, THE Auditor SHALL redirect the user to `/audit/[Audit_ID]`, where the page serves the data that was just persisted.

### Requirement 4: Report Page

**User Story:** As a startup founder, I want to view my audit results on a dedicated shareable page, so that I can review and share my findings.

#### Acceptance Criteria

1. THE Auditor SHALL render the Report_Page at the URL `/audit/[id]` where `[id]` is the Audit_ID.
2. THE Report_Page SHALL display the total monthly spend, total optimized spend, and total Savings each formatted as USD currency with 2 decimal places (e.g., $1,234.56).
3. THE Report_Page SHALL display a per-tool breakdown table showing: tool name, current plan, current monthly cost, Optimized_Plan name, optimized cost, and Savings per tool.
4. IF a tool's Savings is greater than 0, THEN THE Report_Page SHALL display a suggestion for that tool that includes the tool name and the name of the Optimized_Plan being recommended.
5. WHEN the audit record contains a non-empty Summary, THE Report_Page SHALL display the Summary text.
6. WHEN a user navigates to `/audit/[id]` with a valid Audit_ID, THE Auditor SHALL fetch the audit record from Supabase and render the Report_Page.
7. IF the Audit_ID does not exist in Supabase, THEN THE Auditor SHALL render a not-found page with a message indicating the audit was not found.
8. THE Report_Page SHALL include a copy-to-clipboard button that copies the full `/audit/[id]` URL to the user's clipboard and displays a visible confirmation (e.g., "Copied!") after the copy succeeds.
9. IF the Supabase fetch fails for a valid-format Audit_ID, THEN THE Auditor SHALL render an error page with a human-readable message indicating a data retrieval failure.

### Requirement 5: Claude AI Summary Generation

**User Story:** As a startup founder, I want a personalized 100-word summary of my audit, so that I can quickly understand my key savings opportunities.

#### Acceptance Criteria

1. WHEN the Audit_Engine completes calculation for a submitted audit, THE Auditor SHALL call the Anthropic Claude API to generate a Summary.
2. THE Auditor SHALL pass the total spend, total optimized spend, total Savings, and per-tool breakdown as context to the Claude API prompt.
3. THE Auditor SHALL instruct the Claude API to produce a Summary of approximately 100 words (between 90 and 110 words inclusive), written in second person, addressed to a startup founder.
4. WHEN the Claude API returns a successful response, THE Auditor SHALL persist the generated Summary text alongside the audit record in Supabase.
5. IF the Claude API call fails or times out after 30 seconds, THEN THE Auditor SHALL store an empty string as the Summary in Supabase and render the Report_Page without a summary section, without blocking the audit result.
6. IF the Supabase write for the Summary fails independently of the audit record write, THEN THE Auditor SHALL log the failure and render the Report_Page without a summary section.

### Requirement 6: Email Report Delivery

**User Story:** As a startup founder, I want to enter my email address on the results page to receive my audit report by email, so that I have a permanent copy.

#### Acceptance Criteria

1. THE Report_Page SHALL display an email capture form with a single email address input field and a submit button.
2. WHEN a user submits an email address that conforms to RFC 5322 format, THE Auditor SHALL send the audit report via Resend to that email address.
3. THE email SHALL include: the Audit_ID, total spend, total optimized spend, total Savings, per-tool breakdown, the Summary (if non-empty), and a link to the Report_Page URL.
4. WHEN the Resend API call succeeds, THE Auditor SHALL persist the submitted email address alongside the Audit_ID in Supabase.
5. WHEN the email is sent successfully, THE Auditor SHALL display a confirmation message on the Report_Page and disable the email capture form to prevent duplicate submissions.
6. IF the Resend API call fails, THEN THE Auditor SHALL display an error message on the Report_Page, SHALL NOT persist the email address, and SHALL re-enable the email capture form so the user can retry.
7. WHEN a user submits an email address that does not match the RFC 5322 format, THE Auditor SHALL display a validation error and prevent submission.
8. WHEN a user submits an email address that has already been persisted for the same Audit_ID, THE Auditor SHALL display an informational message indicating the report has already been sent to that address and SHALL NOT send a duplicate email.

---

### Requirement 7: Pricing Catalog Data Integrity

**User Story:** As a startup founder, I want the savings calculations to be based on accurate, up-to-date pricing data, so that the recommendations are trustworthy.

#### Acceptance Criteria

1. THE Pricing_Catalog SHALL contain pricing data for all 15 tools listed in Requirement 1, and each tool's key in the Pricing_Catalog SHALL exactly match the corresponding tool name string from the Requirement 1 dropdown (case-sensitive).
2. THE Pricing_Catalog SHALL define at least one plan tier per tool.
3. THE Pricing_Catalog SHALL define a price per seat per month (a number greater than or equal to 0.00 USD) and a minimum seat threshold (a positive integer greater than or equal to 1) for each plan tier of every tool.
4. THE Pricing_Catalog SHALL be defined as a static TypeScript constant, versioned in source control, and not fetched from an external API at runtime.
5. WHEN the Audit_Engine reads the Pricing_Catalog, THE Pricing_Catalog SHALL return consistent results for the same tool name and seat count input (idempotent reads).
6. THE Pricing_Catalog SHALL not contain duplicate plan tier names for the same tool.

---

### Requirement 8: Shareable URL and Navigation

**User Story:** As a startup founder, I want the audit results URL to be shareable with my team, so that others can view the same report.

#### Acceptance Criteria

1. THE Report_Page URL SHALL follow the pattern `/audit/[Audit_ID]` where Audit_ID is a UUID v4.
2. WHEN a user opens a Report_Page URL in a new browser session without having submitted the form, THE Auditor SHALL render the full report (all sections: summary metrics, per-tool breakdown, suggestions, Claude summary, and email capture form) by fetching data from Supabase using the Audit_ID from the URL.
3. THE Report_Page SHALL include Open Graph metadata — title, description, and canonical URL — where the title and description are derived from the audit data (e.g., total savings amount), so that the link renders a meaningful preview when shared on social platforms.
4. IF the Audit_ID in the URL does not exist in Supabase, THEN THE Auditor SHALL render a not-found page with a human-readable message and a link back to the home page.
5. IF the Supabase fetch fails when loading a Report_Page, THEN THE Auditor SHALL render an error page with a human-readable message and a link back to the home page.
